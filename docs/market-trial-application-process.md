# 市场申请客户试用流程方案

## 1. 方案目标

市场在推进客户商机时，经常需要为客户开通试用环境，用于验证具体业务场景、产品能力和服务价值。为避免试用开通过程中出现范围不清、额度不清、客户主体不清的问题，需要统一明确解决方案、套餐和租户三者的职责边界与配合流程。

本方案用于规范市场申请客户试用的完整流程，重点说明：

- 解决方案负责定义客户试用的业务场景和能力范围。
- 套餐负责定义客户试用期间的资源额度和服务边界。
- 租户负责承接客户主体、服务周期、开通状态和后续运营。

## 2. 核心关系

市场申请客户试用时，三者关系应固定为：

```text
解决方案 = 客户试用什么场景
套餐 = 这个场景按什么额度试用
租户 = 给哪个客户实际开通
```

也就是说，试用开通应遵循：

```text
先定场景，再定额度，最后落到租户开通。
```

## 3. 三方职责边界

| 对象 | 负责回答的问题 | 核心内容 | 典型状态或信息 |
|---|---|---|---|
| 解决方案 | 客户试用什么业务场景 | 方案名称、适用场景、模块能力、应用清单、基础配置、交付范围 | 草稿、已发布、停用 |
| 套餐 | 客户按什么资源边界试用 | 可用解决方案、用户数、管理员数、空间数、存储容量、AI 调用、消息额度、试用周期 | 草稿、上架、下架 |
| 租户 | 给哪个客户开通试用 | 客户名称、租户编码、联系人、行业/区域、绑定套餐、服务周期、客户成功负责人 | 试用中、启用、到期、停用、冻结 |

## 4. 完整试用流程

### 4.1 市场提交试用需求

市场在客户试用前，应先提交试用需求，明确客户主体和试用目标。

建议提交信息包括：

| 信息项 | 说明 |
|---|---|
| 客户名称 | 需要开通试用的学校、区域、集团校或企业名称 |
| 客户类型 | 学校、区域、集团校、企业等 |
| 行业/区域 | 用于后续运营分类和客户分析 |
| 联系人 | 客户侧试用负责人 |
| 联系方式 | 电话、微信、邮箱等 |
| 试用目标 | 本次试用希望验证的业务价值 |
| 期望解决方案 | 市场初步判断适合客户的解决方案 |
| 预计用户规模 | 试用期间预计参与人数 |
| 预计试用周期 | 计划开始时间和结束时间 |
| 商机负责人 | 市场侧跟进人 |

本阶段输出：客户试用申请信息。

### 4.2 确认解决方案

产品、方案或交付负责人根据客户需求判断适配的解决方案。

解决方案负责定义：

- 适用业务场景。
- 包含哪些产品模块。
- 哪些应用必须安装。
- 哪些应用可以选配。
- 需要哪些初始化配置。
- 试用期间需要验证的关键流程。

示例解决方案：

| 解决方案 | 适用场景 | 常见模块 |
|---|---|---|
| 教师数字素养提升培训方案 | 教师培训、活动运营、成果认证 | 空间、资料库、Lucky、消息、研讨会、问卷、证书 |
| 区域教研共创解决方案 | 区域教研、共同体协作、资源共建 | 空间、资料库、知识空间、消息 |
| AI 课程创作中心方案 | 课程创作、资源生成、AI 辅助备课 | 空间、资料库、Lucky、知识空间、消息 |

本阶段输出：确认后的解决方案清单。

### 4.3 匹配试用套餐

运营、商务或方案负责人根据客户规模和试用目标选择套餐。

套餐负责定义：

- 允许试用哪些解决方案。
- 成员数量上限。
- 管理员数量上限。
- 部门或组织数量上限。
- 空间数量上限。
- 存储容量上限。
- AI 调用次数或 Token 额度。
- 资料库容量和解析次数。
- 消息、研讨会、问卷、证书等模块额度。
- 是否支持高级能力，例如自定义主题、高级统计等。

建议设置试用套餐类型：

| 套餐类型 | 适用对象 | 使用建议 |
|---|---|---|
| 标准试用套餐 | 单校、小规模客户、早期商机 | 控制用户数和资源额度，适合快速验证 |
| 专业试用套餐 | 区域、集团校、多部门试点 | 支持多个场景组合，适合重点商机 |
| 定制试用套餐 | 战略客户、项目型客户 | 需明确审批人、试用周期、资源上限和退出机制 |

本阶段输出：客户试用所绑定的套餐。

### 4.4 创建或绑定租户

客户没有租户时，创建新租户；客户已有租户时，复用原租户。

租户负责承接：

- 客户主体信息。
- 租户编码。
- 客户联系人。
- 当前状态。
- 绑定套餐。
- 服务开始时间。
- 服务结束时间。
- 客户成功负责人。
- 备注和运营信息。

试用阶段建议状态：

| 状态项 | 建议值 | 说明 |
|---|---|---|
| 租户状态 | 试用中 | 表示客户处于试用阶段 |
| 服务状态 | 试用服务 | 表示当前服务不是正式付费服务 |
| 服务开始时间 | 申请通过后的实际开通日期 | 用于计算试用周期 |
| 服务结束时间 | 明确到具体日期 | 用于到期提醒和试用收口 |
| 客户成功负责人 | 必填 | 用于试用过程中的客户运营和转化跟进 |

本阶段输出：已创建或已绑定的客户租户。

### 4.5 开通试用服务

开通时按“解决方案 + 套餐 + 租户”的结果执行。

开通规则：

- 解决方案决定开哪些模块和应用。
- 套餐决定这些模块能使用多少额度。
- 租户决定这些能力开给哪个客户、开多久、当前处于什么服务状态。

开通检查项：

| 检查项 | 说明 |
|---|---|
| 解决方案是否确认 | 必须明确客户要试用的业务场景 |
| 套餐是否确认 | 必须明确资源额度和试用周期 |
| 租户是否存在 | 必须有承接客户主体的租户 |
| 应用是否安装 | 必装应用应完成安装和初始化 |
| 资源额度是否生效 | 用户数、空间数、存储、AI 调用等限制应可控 |
| 客户负责人是否明确 | 客户侧和内部侧负责人都应明确 |

本阶段输出：客户可访问并可使用的试用环境。

### 4.6 试用过程跟进

试用期间，市场、客户成功、方案和产品需要分工配合。

| 角色 | 主要职责 |
|---|---|
| 市场 | 跟进商机推进、客户反馈和转化意向 |
| 客户成功 | 引导客户使用、组织培训、跟进活跃情况 |
| 方案负责人 | 判断试用场景是否跑通，支持方案调整 |
| 产品/技术支持 | 处理产品问题、配置问题和能力边界问题 |

试用期间重点观察：

- 客户是否完成关键业务流程。
- 实际参与用户是否达到预期。
- 客户是否持续使用核心模块。
- 套餐额度是否满足试用目标。
- 是否出现明显的扩容、升级或定制需求。
- 是否具备转正式条件。

本阶段输出：试用过程记录、客户反馈和转化判断。

### 4.7 定期输出租户试用报告

试用期间应定期输出租户试用报告，用于让市场、客户成功、方案和管理团队及时了解客户真实使用情况，判断是否需要运营介入、资源调整、延期或推进转正式。

报告建议按周输出，重点客户可按需增加专项报告。

| 报告类型 | 输出频率 | 适用对象 | 主要目的 |
|---|---|---|---|
| 周度试用报告 | 每周一次 | 所有试用租户 | 跟踪使用进展、发现风险、推动跟进动作 |
| 到期预警报告 | 到期前 7 到 14 天 | 即将到期租户 | 判断转正式、延期或停用 |
| 重点客户专项报告 | 按商机需要 | 战略客户、重点项目客户 | 支持商务推进、方案复盘和管理决策 |
| 月度试用汇总报告 | 每月一次 | 管理团队 | 复盘整体试用转化、客户质量和资源投入产出 |

租户试用报告建议包含以下内容：

| 报告模块 | 说明 |
|---|---|
| 基本信息 | 租户名称、客户类型、行业/区域、试用套餐、解决方案、服务开始时间、服务结束时间 |
| 使用概览 | 登录人数、活跃人数、管理员使用情况、最近一次活跃时间 |
| 场景进展 | 是否完成关键业务流程，是否使用核心模块，是否达到试用目标 |
| 资源使用 | 用户数、空间数、存储容量、AI 调用、资料库解析、消息、问卷、研讨会、证书等额度使用情况 |
| 客户反馈 | 客户提出的问题、需求、阻塞点、满意度或风险信号 |
| 运营动作 | 本周期已完成的培训、回访、配置支持、问题处理和下一步计划 |
| 转化判断 | 高意向、需培育、风险、无意向等判断，以及对应处理建议 |
| 到期建议 | 转正式、延期试用、停用、冻结或继续观察 |

报告中的关键判断建议统一口径：

| 判断项 | 说明 |
|---|---|
| 使用健康 | 根据登录、活跃、核心流程完成情况判断客户是否真实使用 |
| 场景验证 | 判断客户是否完成了本次试用要验证的业务场景 |
| 资源合理性 | 判断当前套餐额度是否偏低、偏高或需要调整 |
| 转化可能 | 判断客户是否具备进入正式商务推进的条件 |
| 风险等级 | 标记无人使用、负责人缺失、场景不匹配、即将到期未跟进等风险 |

报告输出后的处理要求：

- 市场根据报告判断是否推进商务转化。
- 客户成功根据报告安排培训、回访和使用引导。
- 方案负责人根据报告判断场景是否需要调整。
- 产品或技术支持根据报告处理能力缺口、配置问题和异常问题。
- 对高风险租户，应明确责任人和处理截止时间。
- 对即将到期租户，应在到期前完成转正式、延期或停用决策。

本阶段输出：周度试用报告、到期预警报告、重点客户专项报告或月度试用汇总报告。

### 4.8 试用结束处理

试用到期前，应提前评估结果，避免试用长期悬空。

试用结果分为三类：

| 结果 | 处理方式 |
|---|---|
| 转正式 | 租户状态改为启用，绑定正式套餐，更新正式服务周期和合同信息 |
| 延期试用 | 延长服务结束时间，保留试用状态，记录延期原因和审批结果 |
| 不转化 | 租户停用或冻结，试用服务到期，保留客户和试用记录 |

到期处理要求：

- 每个试用租户必须有明确到期日期。
- 延期必须说明原因，不建议无限期延期。
- 转正式必须重新确认正式套餐、价格、服务周期和交付范围。
- 不转化客户应保留试用记录，便于后续重新激活。

本阶段输出：转正式、延期或停用的处理结果。

## 5. 三方配合矩阵

| 环节 | 解决方案 | 套餐 | 租户 |
|---|---|---|---|
| 申请阶段 | 判断客户适合哪个业务场景 | 判断需要多大资源额度 | 判断是否已有客户主体 |
| 审批阶段 | 确认场景是否可交付 | 确认资源是否合理 | 确认客户信息是否完整 |
| 开通阶段 | 提供模块和应用清单 | 提供资源限制和服务周期 | 承接实际开通对象 |
| 使用阶段 | 保障场景能跑通 | 控制额度和权限边界 | 记录客户状态和服务信息 |
| 到期阶段 | 判断是否完成场景验证 | 判断是否升级或转正式 | 执行启用、延期、停用或冻结 |

## 6. 审批口径

市场申请客户试用时，不建议直接开租户。应先完成三个确认：

```text
客户是否明确？
试用场景是否明确？
套餐额度是否明确？
```

只有以上三项都明确后，才进入租户开通。

建议审批关注点：

| 审批点 | 判断标准 |
|---|---|
| 客户真实性 | 是否有明确客户、联系人和商机负责人 |
| 试用必要性 | 是否有明确试用目标和转化预期 |
| 方案匹配度 | 客户需求是否能被现有解决方案覆盖 |
| 套餐合理性 | 资源额度是否与客户规模和试用目标匹配 |
| 试用周期 | 是否有明确开始和结束时间 |
| 运营责任人 | 是否有人负责试用过程跟进 |

## 7. 推荐数据记录

为了便于后续跟进和复盘，建议每次试用至少保留以下信息：

| 类别 | 字段 |
|---|---|
| 客户信息 | 客户名称、客户类型、行业、区域、联系人、联系方式 |
| 申请信息 | 申请人、申请时间、试用目标、商机阶段、预期转化时间 |
| 方案信息 | 解决方案名称、适用场景、模块清单、应用清单 |
| 套餐信息 | 套餐名称、用户上限、空间上限、存储上限、AI 额度、其他模块额度 |
| 租户信息 | 租户名称、租户编码、租户状态、服务状态、服务开始时间、服务结束时间 |
| 运营信息 | 客户成功负责人、使用记录、客户反馈、延期记录、转化结果 |

## 8. 异常场景处理

| 异常场景 | 处理建议 |
|---|---|
| 客户需求无法匹配现有解决方案 | 不直接开通试用，先由方案负责人评估是否需要定制方案 |
| 市场要求先开租户但场景不明确 | 暂缓开通，补充试用目标和解决方案 |
| 客户规模超过试用套餐额度 | 选择更高试用套餐或走定制试用审批 |
| 试用中途需要增加模块 | 先确认解决方案是否包含该模块，再确认套餐额度是否覆盖 |
| 试用中途资源不够 | 评估是否升级试用套餐、临时扩容或提前转正式 |
| 试用到期无人跟进 | 租户进入到期或冻结状态，并提醒商机负责人处理 |

## 9. 一句话总结

市场申请客户试用时，解决方案管“试什么”，套餐管“能用多少”，租户管“给谁开、开多久、当前是什么状态”。三者边界清晰后，试用开通、过程运营、转正式、延期和停用都会更容易管理。
